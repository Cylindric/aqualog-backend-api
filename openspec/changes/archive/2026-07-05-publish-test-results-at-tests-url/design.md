## Context

The project already runs pytest via Taskfile tasks and now includes coverage reporting in terminal output. There is no browsable HTML artifact exposed by the API server, so reviewing full test details requires shell access and rerunning commands. This change introduces a report generation path and static serving route at `/tests`.

## Goals / Non-Goals

**Goals:**
- Generate pytest HTML report artifacts in a stable directory under the repository.
- Serve generated report files from the running FastAPI service at `/tests`.
- Keep route behavior predictable for both root listing and direct report file access.
- Preserve existing API endpoints and response envelope behavior for API routes.

**Non-Goals:**
- Building a custom test dashboard UI beyond serving generated static HTML.
- Persisting historical report versions with retention policies.
- Exposing test execution controls (triggering tests from HTTP).

## Decisions

- Decision: Use pytest HTML output files generated by task commands.
  - Rationale: Reuses standard pytest ecosystem and avoids custom report rendering.
  - Alternative considered: Creating bespoke JSON-to-HTML reporting. Rejected for unnecessary complexity.

- Decision: Store reports in a dedicated static directory (for example `artifacts/tests/`).
  - Rationale: Provides a stable server mount target and keeps generated files separate from source.
  - Alternative considered: Writing to `tests/` package directory. Rejected to avoid mixing runtime artifacts with test source.

- Decision: Expose directory using FastAPI static file mounting at `/tests`.
  - Rationale: Minimal runtime complexity and direct browser compatibility for HTML plus assets.
  - Alternative considered: Returning report via API JSON envelope. Rejected because HTML assets require static serving semantics.

- Decision: Keep route available only for existing generated files; missing report files return not-found.
  - Rationale: Avoids implicit test execution from request path and preserves operational safety.
  - Alternative considered: Triggering test run when route is requested. Rejected due to runtime cost and side effects.

- `/tests` should be enabled  only in `dev/test`.

- Report filename should be fixed (`index.html`).

## Risks / Trade-offs

- [Risk] Serving test output may expose sensitive implementation details.
  → Mitigation: Document expected usage for trusted environments; consider environment-gated enablement in future changes.

- [Risk] Stale reports may be mistaken for latest test results.
  → Mitigation: Standardize task command usage and output naming conventions in docs.

- [Trade-off] Static serving is simple but not access-controlled by default.
  → Mitigation: Keep current scope to local/dev workflows and add auth controls in a follow-up if needed.

## Migration Plan

- Add or update pytest task commands to emit HTML report artifacts.
- Mount the report artifact directory in FastAPI at `/tests`.
- Add automated tests verifying `/tests` availability and not-found behavior.
- Validate task execution and browser access locally.
- Rollback by removing static mount and reverting task flags.
