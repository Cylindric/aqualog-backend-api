## Context

The project already produces coverage output during task-based test runs, including an HTML coverage directory. That output is not currently exposed through the running API server, so browser review of coverage requires local filesystem access. This change adds a static mount for the coverage HTML output at `/coverage` in development and test environments.

## Goals / Non-Goals

**Goals:**
- Expose the existing HTML coverage report directory through the API server at `/coverage`.
- Keep the coverage report browsable in a normal web browser without requiring terminal access.
- Restrict the route to non-production developer workflows where static report browsing is expected.
- Preserve existing API routes and response envelope behavior.

**Non-Goals:**
- Building a custom coverage dashboard beyond serving generated HTML.
- Triggering test or coverage generation from the HTTP route.
- Adding historical coverage report retention or comparison views.

## Decisions

- Decision: Serve the coverage HTML output directory as static content.
  - Rationale: Coverage already emits HTML artifacts; serving them directly avoids extra report transformation.
  - Alternative considered: Generating a custom API response with coverage data. Rejected because the output is already meant for browser rendering.

- Decision: Use a stable directory generated by the existing coverage task.
  - Rationale: Keeps the task workflow and server mount aligned and predictable.
  - Alternative considered: Writing coverage artifacts into the source tree. Rejected because generated outputs should remain separate from application code.

- Decision: Mount the route only for `dev` and `test` application environments.
  - Rationale: Coverage reports are a developer tool and should not be exposed by default in production.
  - Alternative considered: Always mounting `/coverage`. Rejected for unnecessary exposure of internal test artifacts.

- Decision: Keep `/coverage` behavior independent of API JSON response envelopes.
  - Rationale: Static HTML serving should not affect structured API contracts.
  - Alternative considered: Wrapping coverage files in JSON. Rejected because it would break browser-native viewing.

## Risks / Trade-offs

- [Risk] Static coverage files may expose implementation details if enabled in the wrong environment.
  → Mitigation: Restrict the mount to `dev` and `test`.

- [Risk] Coverage output may become stale if the task is not rerun.
  → Mitigation: Keep coverage generation tied to the existing task workflow and document the browser path.

- [Trade-off] Static serving is simple but does not provide access control.
  → Mitigation: Limit scope to local development and test usage.

## Migration Plan

- Confirm the coverage task writes HTML output to a stable directory.
- Add the `/coverage` static mount in the FastAPI app for `dev` and `test`.
- Add tests covering success, missing-file, and environment-gating behavior.
- Validate the task workflow and browser access locally.
- Roll back by removing the static mount and reverting the task output path.

## Open Questions

- Should the mount serve only the top-level HTML report or the entire coverage output directory tree?
- Do we want `/coverage` to mirror the same path semantics as `/tests`, including environment gating and missing-file behavior?
